# UX/UI Specification: Multi-Tenant Dashboard & Quality Control

**Version:** 1.0 (Draft)
**Target:** `ro-ai-dashboard`
**Date:** 2026-02-21

This document outlines the UX/UI updates required to support the new **Multi-Tenant Architecture** and the **Data Quality Control** pipeline.

---

## 1. Authentication & Multi-Tenant Context

Since a single deployment can now serve multiple domains (e.g., `Gaming`, `Medical`), the dashboard must authenticate the user and clearly indicate which tenant's data is currently being viewed.

### 1.1 Login Page (`/login`)
*   **Purpose:** Secure entry point using On-Premise JWT authentication.
*   **UI Components:** standard email/password form, branding elements.
*   **Logic:** Upon success, Next.js sets a secure HTTP-only cookie with the JWT containing the user's role and associated `tenant_id`s.

### 1.2 Global Tenant Switcher
*   **Location:** Top application header (NavBar).
*   **Component:** A dropdown menu or combo box.
*   **Visibility (RBAC Rule):** 
    *   **Visible only to** `Super Admin` or users assigned to *multiple* tenants.
    *   If a user is only assigned to one tenant, hide the switcher and enforce that tenant statically.
*   **Behavior:** 
    *   Changing the tenant updates a global Context provider (React Context/Zustand).
    *   All API calls (fetching runs, QA data, vectors) automatically append `?tenant_id=<selected_id>`.
    *   The UI theme could slightly shift based on the tenant (e.g., Blue for Medical, Red for Gaming) for visual cues.

## 2. Ingress & Sources Management (New Screen)

Allows admins to manage and configure input sources per tenant.

### 2.1 Screen: `/sources`
*   **Visibility (RBAC Rule):** Restricted to `Super Admin` and `Tenant Admin`.
*   **Purpose:** Configure how data enters the system for the current tenant.
*   **Layout:**
    *   **Header:** "Data Ingress Configurations" + [Add New Source] Button.
    *   **Data Table:** Lists existing sources.
        *   Columns: `Name` | `Type` (Web, Table, PDF) | `Schedule` (Cron) | `Status` | `Actions`.
*   **Add Source Dialog:**
    *   **Step 1 (Type):** Select Source Type (Web Crawler, Upload CSV/XLSX, Upload PDF/Images, MCP Server).
    *   **Step 2 (Detail):**
        *   *If Web:* Enter URL and depth.
        *   *If Table:* File upload drag & drop box. Options for "Has Headers".
        *   *If Documents:* Folder upload.
    *   **Step 3 (MetaData):** Optional `valid_from` and `valid_to` date pickers for the uploaded data.

## 3. Data Quality Control Dashboard (New Screen)

This is the hub for reviewing Q/A Clusters, resolving conflicts, and merging duplicates.

### 3.1 Screen: `/quality_control`
*   **Visibility (RBAC Rule):** Accessible by `Super Admin`, `Tenant Admin`, and `Editor`. Hidden from `Viewer`.
*   **Purpose:** Human-in-the-loop review for the QA pipeline.
*   **Metrics Row (Top):**
    *   Card: Total Clusters
    *   Card: Pending Conflicts (Red/Warning state)
    *   Card: Auto-Merged Duplicates (Green/Success state)
*   **Main View: Kanban / Filterable List**
    *   Two main tabs: **[Action Required (Conflicts)]** | **[Resolved / Consensus]**

### 3.2 Conflict Resolution UI (Detail View)
When an admin clicks on a "Conflict" cluster, they see:
*   **Top Bar:** Cluster Topic / Summary Generated by Gemini.
*   **Side-by-Side Comparison:**
    *   Left Column: Source A (Question + Answer)
    *   Right Column: Source B (Question + Answer)
*   **Gemini Analysis Box (Highlight):** Text explaining *why* there is a conflict (e.g., "Source A says max level is 99, Source B says 150").
*   **Resolution Area (Bottom):**
    *   Option 1: [Accept Source A] (Discards B)
    *   Option 2: [Accept Source B] (Discards A)
    *   Option 3: [Manual Override] (Opens an editor to manually write the "Golden Answer").
    *   [Save & Index to Qdrant] Button.

### 3.3 Golden Answer Suggestion (Auto-Merge UI)
For purely duplicate questions (Consensus):
*   **List View:** Shows the grouped questions.
*   **Golden Answer Box:** Shows the optimized, summarized answer Gemini generated from the duplicates.
*   **Action:** Admins can simply click [Approve] or click on the text to tweak it before approving.

## 4. Vector Database Visualization Updates

### 4.1 Screen: `/vector`
*   **Visibility (RBAC Rule):** Accessible to all authenticated roles (including `Viewer`).
*   **Update:** Add filtering controls for `tenant_id` and validity periods.
*   **Filters:**
    *   Search Box (Semantic Search).
    *   Tenant Dropdown (Linked to Global Switcher).
    *   Toggle: [Show Expired Data] (Filters based on `valid_from` / `valid_to`).
*   **Item Cards:** When displaying a Vector node, show badges for its active period (e.g., `Valid: 1 Jan 2026 - 31 Jan 2026`).

---

## 5. User Management Dashboard (New Screen)

### 5.1 Screen: `/users`
*   **Visibility (RBAC Rule):** Restricted to `Super Admin` and `Tenant Admin`.
*   **Purpose:** Administer the On-Premise user base and assign RBAC roles.
*   **Features:**
    *   Add/Create a new user.
    *   Deactivate user accounts.
    *   Assign users to a `tenant_id` with a specific `role` (Tenant Admin, Editor, Viewer).

---

## 6. Implementation Roadmap (Frontend)

1.  **Phase 1:** Implement Login Screen (`/login`), JWT storage, and Global Tenant Context containing the user's Role.
2.  **Phase 2:** Build RBAC UI Masking (Hiding buttons/pages based on the active role).
3.  **Phase 3:** Build the `/users` management page.
4.  **Phase 4:** Build the `/sources` page with file upload capabilities for Tabular and Media files.
5.  **Phase 5:** Develop the `/quality_control` dashboard, focusing on the Conflict Resolution UI.
6.  **Phase 6:** Update the `/vector` explorer to support Expiration date filtering.
