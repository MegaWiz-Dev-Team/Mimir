# Software Requirements Specification (SRS)
**Project Name:** Mimir Asgard Platform 
**Feature Name:** Asgard Autonomous Agent Ecosystem (RAG & Swarm Orchestration)
**Document Version:** 2.0 (Master Blueprint)
**Date:** 2026-03-31
**Standard:** Aligned with ISO/IEC 29110 Software Implementation (SI) Process

## 1. Introduction

### 1.1 Purpose
This document specifies the software requirements for the comprehensive **Asgard Autonomous Agent Ecosystem**. It details a 4-phase architecture spanning from automated data ingestion (Upstream), to precision RAG tuning (Midstream), Agent Builder deployment with Swarm coordination (Downstream), and finally, a self-improving Meta-Orchestrator (The Overseer).

### 1.2 Scope
The scope encompasses backend data processing agents, the RAG Playground UI/API, the Agent Builder Hub (integrating Soul/Skill decoupling and Swarm patterns), and the background Overseer mechanisms for system self-healing and auto-scaling.

---

## 2. Phase 1: Data Source & Ingestion Pipeline (Upstream)

### 2.1 FR-01: The 5-Agent Ingestion Flow
**Description:** The system shall automatically process raw connected data sources through a sequential 5-agent pipeline via horizontal DAG architecture.
- **Extractor Agent:** Cleans text, standardizes formats, and extracts tabular data.
- **Semantic Splitter Agent:** Chunks documents using semantic context boundaries rather than arbitrary character limits.
- **Graph Architect:** Extracts specialized entities (e.g., Medical codes) and writes relationships into the Neo4j Knowledge Graph.
- **Q&A Synthesizer:** Generates synthetic "Golden Data" questions that a chunk effectively answers.
- **QC Evaluator:** Reviews chunk quality and flags uncertain data for human review.

### 2.2 FR-02: Vector Space Clustering
**Description:** A background job that clusters all document embeddings (Qdrant).
- **Processing:** Applies K-Means/HDBSCAN clustering to semantic vectors.
- **Outputs:** Extracts cluster topics, generating an observable "Knowledge Map" representing existing domain expertise within the database.

### 2.3 FR-03: Dashboard & Human-in-the-Loop QC
**Description:** A dedicated module for Administrators to oversee the ingestion status.
- **Features:** Must display active sync statuses of data sources, metrics of extracted vectors/nodes, and provide an interface to manually Approve/Reject flagged Synthetic QAs generated by FR-01.

---

## 3. Phase 2: RAG Ensemble Playground (Midstream Tuning Lab)

### 3.1 FR-04: Bare-Metal Search Analysis
**Description:** Retrieve raw chunks from Vector (Qdrant), Graph (Neo4j), and PageIndex concurrently without standard LLM generation masks.
- **Outputs:** Display a list of raw data chunks, relevance scores, and visual badges denoting the data source to aid detailed precision tuning.

### 3.2 FR-05: Adjustable Source Weight & Reranker Configuration
**Description:** Adjust algorithmic search rank parameters in real-time.
- **Inputs:** UI sliders representing percentages for Vector, Graph, and Tree (must equal 100%).
- **Processing:** Route documents through selected Reranking Engines (e.g., Weighted Sum or LLM Cross-Encoder).

### 3.3 FR-06: LLM-Assisted Query Optimization (Agent Optimizer)
**Description:** An integrated Agent to optimize user queries before search.
- **Mechanism:** Expand a simple query into 3-5 enriched, domain-specific search prompts to drastically improve data retrieval depth.

### 3.4 FR-07: Stratified Evaluation Set Builder & Batch Benchmarking
**Description:** Construct comprehensive testing datasets (Eval Sets) by intelligently sampling across Vector Clusters (from FR-02) and Graph Topology to prevent biased testing. Execute Batch Benchmarks to calculate global Hit Rate (@K) and Mean Reciprocal Rank (MRR).

### 3.5 FR-08: Sweet Spot Deployment
**Description:** Transfer optimized RAG configuration weights directly into the Agent Builder format upon finalizing benchmark experiments.

---

## 4. Phase 3: Agent Builder Hub (Downstream Production)

### 4.1 FR-09: Soul & Skill Decoupled Architecture
**Description:** The system must decouple an Agent's definition into independent parameters during creation.
- **Soul (Persona):** Predefined behavioral instructions (e.g., 'Empathic Nurse', 'Strict Biller').
- **Skill (Capabilities):** Attached RAG profile weights (FR-08) and MCP integrations.

### 4.2 FR-10: Swarm Handoff Pattern (A2A Routing & Safety)
**Description:** Implement decentralized Agent-to-Agent (A2A) orchestration.
- **Handoff Tool:** Equip agents with a dynamic `handoff_to_agent()` capability, passing Structured Shared Context objects.
- **Safety Loops (Repetitive Detection):** Implement strict detection windows to intercept and terminate destructive Ping-Pong infinite loops between collaborating agents.

### 4.3 FR-11: MCP Tools Integration
**Description:** Allow administrators to toggle Model Context Protocol (MCP) integrations on an Agent profile (e.g., external weather APIs, local EMR FHIR servers).

### 4.4 FR-12: Base Deployment to Bifrost Gateway
**Description:** A single-click feature to package the generated Agent Profile, Access Routes, and Webhooks, formally publishing the Agent endpoint to the Bifrost API Gateway for live client consumption (e.g., OpenEMR widgets).

---

## 5. Phase 4: The Overseer (Meta-Orchestrator & Self-Healing)

### 5.1 FR-13: Vector Overlay Analysis (Knowledge Gap Detection)
**Description:** A background capability where The Overseer analyzes the semantic space.
- **Processing:** Overlays live User Query clusters directly against Document clusters (FR-02) to automatically detect "Knowledge Blind Spots" (high-density user interest regions with missing documentation).

### 5.2 FR-14: Auto-Healing Orchestration (Tuning & Alerting)
**Description:** Continuous health monitoring of deployed agents via A2A Latency, Token Usage, and Thumbs-Down feedback metrics.
- **Mechanism:** If an agent degrades or blind spots are found, The Overseer automatically re-invokes RAG tuning pipelines (FR-05) via API to adjust search parameters entirely without human intervention.

### 5.3 FR-15: Agent Genesis (Auto-Scaling)
**Description:** Advanced autonomy enabling The Overseer to propose or autonomously spawn new Specialist Agents via the Agent Builder API to counter massive spikes in out-of-domain user intent clusters.

---

## 6. Non-Functional Requirements

### 6.1 Performance
- **Parallel Retrieval:** To prevent sequential blocking bottlenecks, data fetch requests to Qdrant, Tree, and Neo4j stores must utilize concurrent background execution (`tokio::join!`).
- **Distributed Memory:** Shared Working Memory across Swarm Agents must utilize rapid in-memory stores (e.g., Redis).

### 6.2 Usability & Interfaces
- Complex ML configurations should feature aesthetic Light-Mode default designs favoring clean data-visualization (Graphs/DAG pipelines).
- Dashboards must provide real-time slider updates and flowchart progression animations to reflect the active Agent states.
